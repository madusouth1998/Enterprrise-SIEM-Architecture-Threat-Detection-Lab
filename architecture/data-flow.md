# SIEM Data Flow Architecture

## Overview

This document describes the flow of security telemetry through the Enterprise SIEM Architecture & Threat Detection Lab.

The environment was built in Microsoft Azure to collect, centralize, and analyze security events generated by an internet-facing virtual machine. Microsoft Sentinel and Azure Log Analytics were used as the core SIEM and log analytics platforms.

The primary data flow was:

Internet Activity → Azure Network Layer → Honeypot VM → Security Telemetry → Log Analytics Workspace → Microsoft Sentinel → KQL Analysis → Detection and Investigation

---

## 1. Internet Activity

The Azure virtual machine was exposed to the public internet within a controlled lab environment.

This allowed the system to receive real-world unsolicited connection and authentication attempts from automated scanners and credential-guessing infrastructure.

During the observation period, the environment recorded more than 52,000 authentication attempts within approximately 30 minutes.

The purpose of exposing the system was not to attack external infrastructure, but to observe and analyze inbound activity directed toward a system owned and controlled by the project author.

---

## 2. Azure Network Layer

Inbound network traffic passed through the Azure networking layer before reaching the virtual machine.

The environment used:

- Azure Virtual Network
- Network Security Group (NSG)
- Public IP addressing
- Network security rules
- Network telemetry

The Network Security Group controlled permitted inbound and outbound traffic to the virtual machine.

Where enabled, network telemetry provided additional visibility into communication patterns involving the monitored system.

---

## 3. Honeypot Virtual Machine

The internet-facing virtual machine acted as the primary telemetry source for the lab.

The host generated security events related to:

- Successful authentication attempts
- Failed authentication attempts
- Source IP addresses
- Targeted user accounts
- Logon activity
- System security events

Windows Security Event Logs provided the authentication telemetry used for threat hunting and detection analysis.

A key event used during the investigation was:

Windows Event ID 4625 — An account failed to log on

This event was used to identify repeated failed authentication attempts and analyze potential brute-force activity.

---

## 4. Log Collection and Centralization

Security telemetry generated by the monitored system was ingested into an Azure Log Analytics Workspace.

The Log Analytics Workspace acted as the centralized repository for collected security data.

Centralizing the logs made it possible to:

- Search large volumes of security events
- Correlate activity across time
- Aggregate authentication attempts
- Identify high-volume source IP addresses
- Develop reusable KQL queries
- Support Microsoft Sentinel analytics

---

## 5. Microsoft Sentinel

Microsoft Sentinel was used as the SIEM platform for the lab.

Sentinel provided the security monitoring layer used to:

- Query collected telemetry
- Perform threat hunting
- Develop detection logic
- Create analytics rules
- Investigate suspicious activity
- Map detection scenarios to MITRE ATT&CK

Microsoft Sentinel operated on top of the Log Analytics Workspace and used the collected telemetry as the basis for security analysis.

---

## 6. KQL Threat Hunting and Analysis

Kusto Query Language (KQL) was used to query and analyze the collected security events.

The analysis process included:

1. Identifying the relevant data source.
2. Filtering for security events of interest.
3. Grouping related events by source and time.
4. Counting repeated authentication attempts.
5. Identifying abnormal patterns.
6. Developing detection logic from the observed behavior.

Example analytical flow:

Raw Security Events
        ↓
Filter Relevant Event IDs
        ↓
Extract Source and Account Information
        ↓
Aggregate Events Over Time
        ↓
Identify Suspicious Patterns
        ↓
Apply Detection Thresholds
        ↓
Investigate Results
